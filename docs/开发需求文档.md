# jinHan-shop 开发需求文档

> 版本：v1.0
> 编写日期：2026-07-21
> 依据：`docs/商品列表功能缺失分析.md`、当前仓库代码审计结果
> 范围：`jinHan-shop-core`、`jinHan-shop-admin-web`、新增 `jinHan-shop-user-web`

---

## 一、项目现状

### 1.1 已交付能力

| 域 | 已交付 |
|----|--------|
| 后台认证 | 登录、验证码、Sa-Token 鉴权 |
| 权限体系 | 管理员、角色、权限、权限批量同步 |
| 基础资料 | 供应商、供应商品牌、商品品牌、商品分类（树） |
| 商品 | SPU 增删改查/分页、SKU CRUD/分页、图片 CRUD/分页、商品详情、审核通过 |
| 基础设施 | 文件上传、配置中心、缓存、MyBatis 分页 |

### 1.2 进行中

- 商品列表排序改造：`ProductSpuPageQueryCommand` 已切换为 `SortablePageParam`（未提交），Mapper XML 动态排序改造中。

### 1.3 主要缺口

1. 商品列表仍是"数据表分页"，缺排序白名单、SKU 聚合、价格/时间筛选、批量操作。
2. 审核流程不闭环：无拒绝入口、无审核记录、编辑已上架商品不触发重审。
3. 状态一致性风险：新建商品可直接上架但审核状态为待审核。
4. 商品详情不满足编辑回显：缺 SKU 列表、轮播图、详情图分组。
5. 无 C 端：`jinHan-shop-user-web` 模块不存在，用户、购物车、订单、支付、物流、售后域全部缺失。

## 三、第一阶段：商品列表运营能力（P0）

### REQ-PROD-01 商品列表动态排序

- **背景**：当前 SQL 硬编码 `p.sort ASC, p.created_time DESC`，前端无法按创建时间、销量、价格排序。
- **需求**：
  - `ProductSpuPageQueryCommand` 继承 `SortablePageParam`（进行中）。
  - 后端维护排序字段白名单映射：`createdTime → p.created_time`、`updatedTime → p.updated_time`、`salesCount → p.sales_count`、`sort → p.sort`；REQ-PROD-02 完成后追加 `minPrice`、`totalStock`。
  - 白名单外的字段直接忽略或报错，禁止透传至 SQL。
  - 无排序参数时保持现有默认排序。
- **验收标准**：
  - 传入 `sortingFields=[{field:"createdTime",order:"desc"}]` 返回按创建时间倒序。
  - 传入非法字段名不产生 SQL 注入、不影响查询。
- **涉及文件**：`ProductSpuPageQueryCommand`、`ProductSpuPageQueryHandler`、`ProductSpuMapper.xml`。

### REQ-PROD-02 列表 SKU 聚合字段

- **背景**：运营在列表页看不到价格区间、库存、SKU 数。
- **需求**：列表 SQL 按 `spu_id` 聚合 `product_sku`（仅 `deleted=0`），返回：
  - `minPrice`、`maxPrice`（仅统计启用状态 SKU）
  - `totalStock`（全部未删除 SKU 库存合计）
  - `skuCount`、`enabledSkuCount`
- **验收标准**：新建 1 个 SPU + 3 个 SKU（2 启用 1 禁用），列表返回 `skuCount=3`、`enabledSkuCount=2`、价格区间与库存合计正确。
- **涉及文件**：`ProductSpuMapper.xml`、`ProductSpuPageQueryResult`、`ProductSpuPageResponse`。

### REQ-PROD-03 价格区间与时间范围筛选

- **需求**：
  - 入参新增 `minPrice`、`maxPrice`：按 SKU 价格过滤（EXISTS 子查询，仅未删除 SKU）。
  - 入参新增 `createdTimeStart/End`、`updatedTimeStart/End`。
  - 入参新增 `includeChildCategory`：为 true 时查出分类树下全部子分类 ID，按 `category_id IN (...)` 过滤。
- **验收标准**：各筛选条件单独与组合使用结果正确；价格边界值（等于 min/max）包含在内。

### REQ-PROD-04 批量上下架

- **需求**：
  - 新增 `PUT /product/batch/sale-status`，入参：`ids`、`saleStatus`。
  - 上架校验：仅 `auditStatus=PASS` 的商品允许上架，未通过的在响应中逐条列出失败原因。
  - 操作记录操作人（为 REQ-PROD-10 变更日志预留）。
- **验收标准**：批量上架含待审核商品时，待审核商品上架失败并返回明细，其余成功。
- **权限**：新增权限码，登记到 `AdminPermissionConst`。

### REQ-PROD-05 批量删除

- **需求**：新增 `DELETE /product/batch`，事务内级联逻辑删除 SPU、SKU、图片，与单条删除语义一致；已上架商品禁止删除或先自动下架（需产品确认，建议禁止删除并提示）。
- **验收标准**：批量删除后 SPU/SKU/图片 `deleted=1`；任一失败整体回滚。

### REQ-PROD-06 商品详情编辑回显

- **需求**：详情接口 `GET /product/{id}` 扩展返回：
  - SKU 列表（含规格、价格、库存、状态）
  - 图片按 `imageType` 分组：主图、轮播图、详情图
  - `categoryPath` 分类路径（如 `一级 > 二级 > 三级`）
- **验收标准**：前端编辑页仅凭该接口完成全部回显，无需额外请求。

## 四、第二阶段：商品生命周期闭环（P1）

### REQ-PROD-07 审核拒绝

- **背景**：`AuditStatusEnum.REJECT` 已存在但无业务入口。
- **需求**：
  - 新增 `PUT /product/audit/reject/{id}`，入参含必填 `rejectReason`。
  - 状态流转校验：仅 `PENDING` 可拒绝；已 `PASS` 商品需先下架再驳回（或禁止，需产品确认）。
  - 拒绝原因落库并随详情返回。
- **涉及文件**：`ProductController`、`ProductAuditHandler`、新增 `ProductAuditRejectCommand`。

### REQ-PROD-08 新建商品状态一致性

- **需求**：
  - 创建商品强制 `saleStatus=OFF_SHELF`、`auditStatus=PENDING`，忽略入参中的上架状态。
  - C 端任何查询必须同时过滤 `ON_SHELF + PASS`（为 REQ-MALL-03 前置约束）。
- **验收标准**：创建时传 `saleStatus=ON_SHELF`，落库仍为下架待审核。

### REQ-PROD-09 编辑后重新审核

- **需求**：关键字段（名称、分类、品牌、供应商、主图）变更后：
  - `auditStatus` 重置为 `PENDING`；
  - 若商品处于上架状态，自动下架或保持上架但需二次审核（二选一，建议自动下架）。
- **同步修复**：`ProductSpuUpdateHandler` 补齐与创建一致的供应商/分类/品牌存在性校验；清理未使用 import。

### REQ-PROD-10 审核记录与变更日志

- **需求**：
  - 新增 `product_audit_record` 表：商品 ID、动作（通过/拒绝/提交）、操作人、原因、时间。
  - 新增 `product_change_log` 表（或合并设计）：记录上下架、改价、编辑重审等关键动作。
  - 后台提供按商品查询记录的接口。
- **验收标准**：每次审核/上下架操作均可追溯操作人与时间。

### REQ-PROD-11 商品复制

- **需求**：新增 `POST /product/copy/{id}`：复制 SPU + SKU + 图片，生成新 SPU/SKU 编码，复制产物默认 `OFF_SHELF + PENDING`。
- **验收标准**：复制品与原商品数据一致（编码除外），状态为下架待审核。

## 五、第三阶段：C 端商城基础（P2）

### REQ-MALL-01 用户域与 user-web 模块

- **需求**：
  - 新增模块 `jinHan-shop-user-web`，包前缀 `com.jinHan.shop.api`（遵循既有约定）。
  - 新增用户域：用户注册（手机号+短信验证码）、登录、登出、用户信息查询/修改。
  - 用户表设计：手机号唯一、昵称、头像、状态。
  - 鉴权沿用 Sa-Token，与后台 admin 会话隔离。
- **前置**：短信验证码能力（当前仅有图形验证码，需接入短信通道或先用模拟通道）。

### REQ-MALL-02 收货地址

- **需求**：地址 CRUD、默认地址设置；字段含收货人、手机号、省市区、详细地址。

### REQ-MALL-03 C 端商品列表与详情

- **需求**：
  - `GET /api/product/page`、`GET /api/product/{id}`。
  - 强制过滤：`deleted=0 AND saleStatus=ON_SHELF AND auditStatus=PASS AND EXISTS(启用且库存>0 的 SKU)`。
  - 查询模型独立于后台：关键词、分类（含子分类）、品牌、价格区间；排序支持综合、销量、新品、价格升降。
- **验收标准**：未审核/已下架/无可售 SKU 商品在任何筛选组合下均不出现。

### REQ-MALL-04 C 端分类与品牌导航

- **需求**：分类树接口（仅启用分类）、分类下品牌列表、首页推荐位数据结构预留。

### REQ-MALL-05 购物车

- **需求**：购物车增删改查、勾选/取消勾选、数量修改（校验 SKU 状态与库存）、失效商品标记。
- **存储**：登录用户落库（`cart` 表），暂不支持游客购物车。

## 六、第四阶段：交易链路（P2）

### REQ-TRADE-01 订单域

- **需求**：
  - 新增 `order` 子域（`com.jinHan.shop.core.order.domain`）。
  - 下单：购物车/直接购买 → 确认订单（价格快照、库存校验、地址快照）→ 创建订单。
  - 订单模型：订单主表 + 订单明细 + 状态机（待支付/待发货/待收货/已完成/已取消/售后中）。
  - 库存策略：下单扣减或支付扣减需产品确认（建议支付成功扣减 + 下单预占）。
  - 超时未支付自动取消（RabbitMQ 延迟消息或定时任务）。

### REQ-TRADE-02 支付

- **需求**：支付通道接入（微信/支付宝，需商务确认）、支付回调、支付状态同步、退款接口预留。黄金价格波动场景需确认是否支持锁价。

### REQ-TRADE-03 物流发货

- **需求**：后台发货（快递公司、单号）、C 端物流状态查看；贵重品需确认是否强制保价/签收凭证。

### REQ-TRADE-04 确认收货与完成

- **需求**：用户主动确认收货、超时自动确认（如发货后 15 天）、订单完成状态流转。

### REQ-TRADE-05 售后

- **需求**：仅退款/退货退款申请、后台审核、退款执行、售后状态机。黄金类目通常有特殊退货规则，需法务/运营确认后细化。

## 七、第五阶段：运营效率（P3）

### REQ-OPS-01 商品导出

- **需求**：商品列表按当前筛选条件导出 Excel；先修复 `PageParam.PAGE_SIZE_NONE=-1` 与 `@Min(1)` 校验冲突，导出走独立通道绕开分页上限。

### REQ-OPS-02 批量审核

- **需求**：`PUT /product/batch/audit/pass`、`PUT /product/batch/audit/reject`，批量拒绝必须填写统一拒绝原因。

### REQ-OPS-03 低优先代码修复

- `PageParam` 的 `@Max(1000)` 文案 "每页条数最大值为 100" 与数值不一致。
- `ProductSpu` 类注释 `ProductSup` 笔误。
- 主图唯一性：增加 `(spu_id, image_type, deleted)` 唯一约束或查询侧限一条，防止列表 SPU 重复行。

## 十、实施顺序建议

1. **REQ-PROD-01~03**（列表查询模型）→ 衔接当前未提交的排序改造，一次提交。
2. **REQ-PROD-08/07/09**（状态一致性 + 审核闭环）→ 先堵数据风险。
3. **REQ-PROD-04/05/06**（批量操作 + 详情回显）。
4. **REQ-PROD-10/11**（审核记录、商品复制）。
5. **REQ-MALL-01~05**（C 端基础，可与 1~4 并行启动模块搭建）。
6. **REQ-TRADE-01~05**（交易链路，依赖 Q4/Q5 确认）。
7. **REQ-OPS-01~03**（运营效率，随时穿插）。
