# 商品代码审查与商品列表功能缺失分析

> 审计日期：2026-07-04  
> 审计范围：`jinHan-shop/jinHan-shop-core`、`jinHan-shop/jinHan-shop-admin-web` 中商品 SPU、SKU、图片、详情、审核相关代码。  
> 结论：后台商品 SPU 列表已有基础分页和基础筛选，但仍偏“数据表分页”，距离运营可用的商品列表还缺排序、价格库存聚合、价格/时间筛选、批量操作、审核拒绝、C 端可见列表等关键能力。

## 二、商品代码审查结论

### 高风险问题

| 问题 | 影响 | 位置 | 建议 |
|------|------|------|------|
| 商品列表排序被 SQL 固定为 `p.sort ASC, p.created_time DESC` | 前端无法按创建时间、销量、价格、库存等维度排序；`SortablePageParam` 和 `MyBatisUtils.buildPage` 未复用 | `ProductSpuPageQueryCommand.java:17`、`ProductSpuPageQueryHandler.java:27`、`ProductSpuMapper.xml:79` | `ProductSpuPageQueryCommand` 改继承 `SortablePageParam`，由后端维护排序字段白名单，列表 SQL 去掉硬编码排序或改为安全动态排序 |
| 商品审核只有“通过”，没有“拒绝”入口 | `AuditStatusEnum.REJECT` 已存在但业务无法落库，审核流程不闭环 | `AuditStatusEnum.java:16`、`ProductController.java:49`、`ProductAuditHandler.java:28` | 新增 `ProductAuditRejectCommand`、`ProductAuditRejectHandler` 或在现有 Handler 增加 `reject` 方法，接口建议 `PUT /product/audit/reject/{id}` |
| 已审核通过商品被编辑后不会重新进入待审核 | 已上架商品被改名、改分类、改品牌、改供应商后仍可能保持 `PASS`，审核状态无法反映最新内容 | `ProductSpuUpdateHandler.java:45-56` | 明确规则：关键字段变更后重置为 `PENDING`，或禁止编辑已通过/已上架商品，或增加二次审核流 |
| 新增商品可直接传入上架状态，但审核状态固定待审核 | 可能出现 `saleStatus=ON_SHELF` 且 `auditStatus=PENDING` 的不一致状态；若未来 C 端列表只看销售状态，会误展示未审核商品 | `ProductSpuCommand.java:55`、`ProductSpuCreateHandler.java:77-78` | 创建时默认下架，审核通过后再允许上架；C 端列表必须同时过滤 `ON_SHELF + PASS` |
| SPU 编辑只校验 SPU 编码，不校验新供应商、分类、品牌是否存在 | 编辑可写入不存在的外键 ID，导致列表关联名称为空、数据质量下降 | `ProductSpuCreateHandler.java:48-64`、`ProductSpuUpdateHandler.java:45-56` | 编辑逻辑补齐与创建一致的供应商、分类、品牌存在性校验 |

### 中风险问题

| 问题 | 影响 | 位置 | 建议 |
|------|------|------|------|
| 列表返回缺少 SKU 维度聚合 | 运营在列表页看不到最低价、最高价、总库存、SKU 数、启用 SKU 数 | `ProductSpuPageResponse.java:72-102`、`ProductSpuPageQueryResult.java:34-37`、`ProductSpuMapper.xml:27-80` | 列表 SQL 聚合 `product_sku`，返回 `minPrice`、`maxPrice`、`totalStock`、`skuCount`、`enabledSkuCount` |
| 商品详情只返回 SPU 基本信息和主图 | 详情接口无法一次拿到 SKU 列表、轮播图、详情图，不满足商品编辑回显 | `ProductResponse.java:55-75`、`ProductDetails.java:97-100`、`ProductSpuMapper.xml:103-132` | 详情模型增加 SKU 列表和图片分组，或增加组合详情接口 |
| `PageParam.PAGE_SIZE_NONE = -1` 与参数校验冲突 | 注释说明支持 `size=-1` 全量查询，但 `@Min(1)` 会拦截导出场景 | `PageParam.java:23-31`、`BaseMapperX.java:41-48` | 如果要支持导出，需调整校验或导出接口绕开普通分页参数 |
| 排序字段基础设施没有字段白名单 | 如果直接把前端字段透传给 `MyBatisUtils.buildPage`，存在排序字段越权或 SQL 结构风险 | `SortingField.java:31-35`、`MyBatisUtils.java:36-40` | 商品列表单独定义允许排序字段映射，如 `createdTime -> p.created_time`、`salesCount -> p.sales_count` |
| 图片主图关联依赖业务唯一性，没有数据库层约束证据 | 如果同一 SPU 出现多条主图记录，列表会重复 SPU 行 | `ProductSpuMapper.xml:53-56`、`ProductSpuImageCreateHandler` | 增加唯一约束 `(spu_id, image_type, deleted)` 或查询侧取一条主图 |

### 低风险问题

| 问题 | 影响 | 位置 | 建议 |
|------|------|------|------|
| `ProductSpuUpdateHandler` 存在未使用 import | 代码清洁度问题 | `ProductSpuUpdateHandler.java:10`、`ProductSpuUpdateHandler.java:15` | 删除 `SaleStatusEnum`、`Arrays` 未使用导入 |
| `PageParam` 的 `@Max(value = 1000, message = "每页条数最大值为 100")` 文案与数值不一致 | 校验提示误导 | `PageParam.java:31` | 统一为 1000 或把限制改回 100 |
| 商品 SPU 模型注释类名写成 `ProductSup` | 文档注释错误 | `ProductSpu.java:11` | 修正为 `ProductSpu` |

## 四、商品列表缺失功能

### P0：先补齐运营列表基础能力

| 缺失功能 | 当前问题 | 建议实现 |
|----------|----------|----------|
| 动态排序 | 只能固定按排序值和创建时间排序 | 支持 `createdTime`、`updatedTime`、`salesCount`、`sort`，SKU 聚合完成后再支持 `minPrice`、`totalStock` |
| SKU 聚合字段 | 列表无法看到价格、库存、SKU 数 | `product_sku` 按 `spu_id` 聚合，返回 `minPrice`、`maxPrice`、`totalStock`、`skuCount`、`enabledSkuCount` |
| 价格区间筛选 | 入参没有 `minPrice`、`maxPrice` | 用 `EXISTS` 或聚合子查询过滤 SKU 价格，注意只统计未删除 SKU |
| 时间范围筛选 | 入参没有创建/更新时间范围 | 增加 `createdTimeStart`、`createdTimeEnd`、`updatedTimeStart`、`updatedTimeEnd` |
| 批量上下架 | 目前只有单条 SPU 编辑 | 新增批量销售状态接口，校验只允许审核通过商品上架 |
| 批量删除 | 目前只有单条删除 | 新增批量删除 Handler，事务内删除 SPU、SKU、图片 |

### P1：补齐商品生命周期

| 缺失功能 | 当前问题 | 建议实现 |
|----------|----------|----------|
| 审核拒绝 | 枚举有拒绝态，但接口没有拒绝动作 | 新增拒绝接口，建议保留拒绝原因字段 |
| 审核记录/变更历史 | 无法追踪谁审核、为何拒绝、何时改价/上下架 | 设计 `product_audit_record` 或 `product_change_log` |
| 编辑后重新审核 | 通过商品编辑后状态不变 | 关键字段变更后重置审核状态，或限制已上架商品直接编辑 |
| 商品复制 | 无法基于老商品快速复制 SPU、SKU、图片 | 新增 `POST /product/copy/{id}`，复制后生成新编码且默认下架待审核 |

### P2：补齐用户体验和前台能力

| 缺失功能 | 当前问题 | 建议实现 |
|----------|----------|----------|
| 分类树筛选 | 父分类不能查出子分类商品 | 查询分类树后使用 `category_id IN (...)` |
| 分类面包屑 | 列表只有当前分类名，没有完整路径 | 返回 `categoryPath`，如 `一级 > 二级 > 三级` |
| C 端商品列表 | 当前仓库只看到 `jinHan-shop-admin-web`，没有 `jinHan-shop-user-web` | 新增用户端商品浏览接口，只暴露 `ON_SHELF + PASS + 有启用 SKU` 的商品 |
| C 端搜索排序 | 后台列表不等于用户列表 | 用户端需要关键词、分类、品牌、价格、销量、新品等独立查询模型 |
| 导出 | 没有商品导出接口 | 新增导出接口，修复 `PAGE_SIZE_NONE` 校验冲突后复用查询 |

## 六、推荐接口设计

### 后台商品列表

`GET /product/page`

建议新增查询参数：

| 参数 | 说明 |
|------|------|
| `minPrice` / `maxPrice` | SKU 价格区间 |
| `createdTimeStart` / `createdTimeEnd` | 创建时间范围 |
| `updatedTimeStart` / `updatedTimeEnd` | 更新时间范围 |
| `includeChildCategory` | 分类筛选是否包含子分类 |
| `sortingFields` | 排序字段列表，后端做白名单映射 |

建议新增返回字段：

| 字段 | 说明 |
|------|------|
| `minPrice` / `maxPrice` | SKU 最低价/最高价 |
| `totalStock` | SKU 总库存 |
| `skuCount` | SKU 总数 |
| `enabledSkuCount` | 启用 SKU 数 |
| `categoryPath` | 分类路径 |

### 后台批量操作

| 方法 | 路径 | 说明 |
|------|------|------|
| `PUT` | `/product/batch/sale-status` | 批量上下架 |
| `DELETE` | `/product/batch` | 批量删除 |
| `PUT` | `/product/batch/audit/pass` | 批量审核通过 |
| `PUT` | `/product/batch/audit/reject` | 批量审核拒绝 |
| `POST` | `/product/copy/{id}` | 复制商品 |

### 用户端商品列表

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/product/page` | 用户端商品列表 |
| `GET` | `/api/product/{id}` | 用户端商品详情 |

用户端列表默认过滤条件：

```sql
p.deleted = 0
AND p.sale_status = 1
AND p.audit_status = 1
AND EXISTS (
    SELECT 1
    FROM product_sku sku
    WHERE sku.spu_id = p.id
      AND sku.deleted = 0
      AND sku.status = 1
      AND sku.stock > 0
)
```

## 七、建议实施顺序

1. 先补齐 SPU 列表查询模型：排序、价格区间、时间范围、SKU 聚合字段。
2. 修复商品生命周期：创建默认下架、审核拒绝、编辑后审核状态规则、上架前校验审核通过。
3. 增加批量操作：批量上下架、批量删除、批量审核。
4. 扩展商品详情：返回 SKU 列表、轮播图、详情图，满足编辑回显。
5. 新增用户端商品列表与详情接口，严格过滤未审核、下架、无可售 SKU 商品。
6. 最后补充导出、变更历史、分类面包屑、商品复制等运营效率功能。
